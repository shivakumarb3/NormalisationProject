Normalisation

A table is a relation in RDBMS.Normalization is a database design technique which organizes tables in a manner that reduces redundancy and dependency of data. what is the need of normalisation?
Normalisation is needed to avoid anomolies while perfroming operation on databases like write and delete.
consider a relation R(A,B,C,D), if A retrieves the B and c retrieves D(i,e A->B, C->D). If we want to insert the data of A,B we need to insert the data C and D similarly if we want to delete the data (A,B) we need to delete the record C,D unnecessarily. Redundancy of data may also be present because of this.(Every time we insert A,B we need to insert C,D and either C or D can be same for two or more attribute value of A)

There are mainly 4Normal forms highly recommended to Know They are:
  • 1NF
  • 2NF
  • 3NF
  • BCNF

1NF

-All the attribute should be single valued not multivalued
eg:Take an attribute called "phone number" a person can have one or more phone numbers, so just divide the phone number attribute into a sepcial or new table and take primary key of original table as referene key for new table.

2NF

- It should be in 1NF - There should not be partial dependecies. consider above R(A,B,C,D) and functional dependencies( A->B, C->D). The candidate key will be AC and to have no partial dependencies there should not be any dependency such that it is proper subset to candidate key and retrieving non prime attributes. here we have C->D, here C is the proper subset of AC and retrieving non prime attribute D, So it is not in 2nd NF to make it into 2NF divide the table R(A,B) and R(C,D)

3NF

- It should be in 2NF - There should not be any transitive relation. Eg: A->B, B->c then A->C, this type of dependicies should not be present in the relation.
To say a relation in 3NF, if there is a dependency A->B, either 'A' should be a candidate key/superkey or B should be prime attribute. in above relation after divison we have A->B here A is a primary key, satisfying our rule, C->D c is a candidate key as well. So it is in 3NF.

BCNF

- It should be in 3NF -Every functional dependency should depend totally on a candidate key/superkey. Eg: if A->B is there, A must be a candidate key/superkey. It is also called strict 3NF.
